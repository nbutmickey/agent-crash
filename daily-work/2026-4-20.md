将原有的通用 Agent Loop Demo 重新设计为**健康管理 Agent**，并在工程化、安全性、架构范式三个维度上全面升级。

## 一、系统提示词重设计

**问题**：原提示词将工具清单写入 system prompt，破坏 prompt cache，且没有领域约束。

**方案**：
- System prompt 完全静态化，工具清单从 prompt 中移除（模型通过 schema 读取）
- 日期通过新增 `getDate` 工具动态获取，不注入 system prompt
- 针对健康管理场景写入：数据目录约定、日志 JSON 格式、三种角色行为准则、安全红线

**关键决策**：`getDate` 工具返回值仅出现在 messages 历史（动态层），system prompt 永久可缓存。

## 二、工具 Hook 系统

在 `executeTool` 外层添加前后置 Hook 中间件。

### Pre-hooks（执行前拦截）

| Hook | 规则 |
|------|------|
| `writeScope` | write/edit 路径必须在 `health-data/` 内 |
| `bashGuard` | 黑名单关键词（rm、curl、sudo 等）拦截 |
| `readBeforeWrite` | 覆盖已有文件前必须先 `read`，防止覆盖历史记录 |

### Post-hooks（执行后处理）

| Hook | 规则 |
|------|------|
| `auditLog` | write/edit/bash 操作追加写入 `health-data/.audit.log` |
| `sanitizeWebOutput` | 网络内容加 `[UNTRUSTED CONTENT]` 标记，防 prompt injection |
| `validateHealthJson` | 写入健康日志时验证 JSON schema |

## 三、回滚与 SAGA 补偿系统

**设计原则**：只读操作不记录；可逆操作快照回滚；不可逆操作 SAGA 补偿。

### 三种策略

| 工具 | 策略 | 回滚效果 |
|------|------|---------|
| `write`（新建） | snapshot | 删除创建的文件 |
| `write`（覆盖） | snapshot | 精确还原原始内容 |
| `edit` | snapshot | 精确还原原始内容 |
| `TodoWrite` | snapshot | 还原内存 todos 数组 |
| `bash` | saga | diff 还原被改文件，删除新建文件 |
| 只读工具 | none | 不记录 |

### 执行流程

```
executeTool
  ├── runPreHooks()
  ├── captureSnapshot()     执行前按策略快照
  ├── executeToolInner()
  ├── runPostHooks()
  └── rollbackManager.record()
```

Agent 可主动调用 `rollback({ steps: -1 })` 触发全量 SAGA 补偿。

## 四、Agent Loop 架构升级：ReAct → Plan → Execute → Reflect

**问题**：原 ReAct 循环无规划、无反思、无约束，复杂任务容易跑偏或陷入循环。

### 新架构流程

```
PLAN（粗粒度 5-10 步，不挂工具，纯 JSON 输出）
  ↓
for each step:
  EXECUTE（ReAct 微循环）
    约束①：MAX_ITERATIONS = 5
    约束②：TOOL_TIMEOUT = 30s
    约束③：重复调用检测（同 tool+args → 中止）
  step 失败 → 追加 planContext → 回 PLAN（最多 3 次）
  ↓
所有步骤完成
  ↓
REFLECT（仅触发一次，不挂工具，纯 JSON 评估）
  → "done"        : 结束
  → "back_to_plan": 回 PLAN
  ↓
超过最大尝试 → gracefulExit（告知用户失败原因，不崩溃）
```

### 关键设计决策

- Reflect **仅在全部步骤完成或质量不达标时触发**，不逐步触发
- Plan 粗粒度，把不确定性留给 ReAct
- **没有独立的 replan 概念**，step 失败直接回 PLAN
- 自然恢复：每层返回结构化 `StepResult`，不抛异常

## 五、代码工程化拆解

**问题**：原 `agent-loop.ts` 和 `tools.ts` 各自超 300 行，职责混杂。

### 最终目录结构

```
lib/
  agent-loop.ts          54 行，纯编排
  prompts.ts             三个 system prompt（PLAN/EXECUTE/REFLECT）
  rollback.ts            RollbackManager + SAGA helpers
  types.ts               所有共享类型
  hooks/
    pre.ts               Pre-hooks
    post.ts              Post-hooks
  tools/
    registry.ts          ToolRegistry（注册接口）
    context.ts           ToolContext（共享状态）
    sandbox.ts           ROOT + sandboxPath
    index.ts             createTools，组装所有模块
    file.ts              read, write, edit, grep, glob
    shell.ts             bash
    todo.ts              TodoWrite
    date.ts              getDate
    web.ts               webSearch, webFetch
    rollback-tool.ts     rollback
  phases/
    plan.ts              planPhase
    execute.ts           executeStep
    reflect.ts           reflectPhase
```

### 工具注册方式

```ts
registry.register({
  tool: { name: "xxx", description: "...", parameters: ... },
  rollbackStrategy: "snapshot",
  captureRollback: async (args, ctx) => { /* 快照逻辑 */ },
  execute: async (args, ctx) => { /* 执行逻辑 */ },
});
```

**扩展新工具只需在对应文件 `register` + `tools/index.ts` 一行调用，其余代码零改动。**

## 六、工具注入策略讨论

**讨论结论**：

- 工具定义通过 API `tools` 参数传入，**不存在于 messages 数组**，不随 session 累积
- 真正随 session 增长的是 `tool_call` / `tool_result` 消息历史
- 动态注入破坏 prompt cache，且 Plan 阶段已承担语义路由职责
- **当前决策**：全量注入工具到上下文，待工具数量增长到 20+ 时再考虑 Plan-guided 动态注入

**未来演进路径**：
- 20 个以内：全量注入
- 20–50 个：用 `tool_hints` 过滤注入
- 50 个以上：embedding 检索 + Plan 路由双层过滤
